# M28I–M28L Human Policy Engine + Override Board

First-draft human policy and override system for local operator governance: what the agent may do automatically vs require approval, project/pack-level policy, overrides for routing/planning/delegation/execution, and a single review/override board.

---

## 1. Exact CLI usage

All commands are under `workflow-dataset policy` (or `workflow-dataset policy <subcommand>`).

| Command | Purpose |
|--------|---------|
| `workflow-dataset policy show` | Show current policy config (path + JSON). |
| `workflow-dataset policy evaluate --action <action_class> [--project <id>] [--pack <id>]` | Evaluate policy for an action and context. |
| `workflow-dataset policy override --scope <global\|project\|pack> --id <scope_id> --rule <key> [--value true\|false] [--reason ...] [--expires <ISO>]` | Apply a temporary override. |
| `workflow-dataset policy revoke --id <override_id>` | Revoke an override by id. |
| `workflow-dataset policy board [--project <id>] [--pack <id>]` | Override board: active effects + overrides + revoke hints. |
| `workflow-dataset policy explain-blocked --action <action_class> [--project <id>] [--pack <id>]` | Explain why an action is blocked. |
| `workflow-dataset policy explain-allowed --action <action_class> [--project <id>] [--pack <id>]` | Explain why an action is allowed. |

**Options (common):** `--repo-root <path>` on any command to set repo root (default: detected or cwd).

**Example:**

```bash
workflow-dataset policy show
workflow-dataset policy evaluate --action execute_trusted_real --project founder_case_alpha
workflow-dataset policy override --scope project --id founder_case_alpha --rule manual_only --value false --reason "temporary pilot"
workflow-dataset policy board --project founder_case_alpha
workflow-dataset policy revoke --id ov_<id>
workflow-dataset policy explain-blocked --action execute_trusted_real --project founder_case_alpha
```

## 2. Sample operator policy (JSON)

Config is stored at `data/local/human_policy/policy_config.json`. Default (when file is missing) is generated by the store with safe defaults. Example structure:

```json
{
  "action_class_policies": [
    {"action_class": "execute_simulate", "allow_auto": false, "require_approval": true, "allow_batch": true},
    {"action_class": "planner_compile", "allow_auto": false, "require_approval": true, "allow_batch": true},
    {"action_class": "execute_trusted_real", "allow_auto": false, "require_approval": true, "allow_batch": false},
    {"action_class": "executor_resume", "allow_auto": false, "require_approval": true, "allow_batch": false},
    {"action_class": "delegate_goal", "allow_auto": false, "require_approval": true, "allow_batch": false},
    {"action_class": "use_worker_lane", "allow_auto": false, "require_approval": true, "allow_batch": false},
    {"action_class": "routing", "allow_auto": true, "require_approval": false, "allow_batch": false}
  ],
  "approval_defaults": {
    "scope": "global",
    "always_manual": true,
    "may_batch_for_risk": "low"
  },
  "delegation_default": {"scope": "global", "may_delegate": false},
  "routing_overrides": [],
  "blocked_actions": [],
  "exception_policy": {"allow_exceptions": false, "granted_by_scope": "global", "expiry_hours": 24},
  "project_simulate_only": {"founder_case_alpha": true},
  "pack_may_override_defaults": {}
}
```

Overrides are in `data/local/human_policy/overrides.json` (array of override records with `override_id`, `scope`, `scope_id`, `rule_key`, `rule_value`, `reason`, `created_at`, `expires_at`, `revoked_at`).

## 6. Mission control integration

The mission control report includes a **[Human policy]** section when state is collected:

- `active_restrictions_count`, `active_overrides_count`, `override_ids` (up to 5), and a hint: `policy board | policy evaluate --action <action_class> --project <id>`.

State is built in `mission_control/state.py` (human_policy block) from `list_active_effects` and `list_overrides`; current project from `project_case.store.get_current_project_id` when available.

## 7. Remaining gaps (for later refinement)

- **Precedence and merging**: Explicit precedence (e.g. project over global, override over config) is applied in evaluation but not yet documented as a formal precedence table.
- **Expiry enforcement**: Override records may have `expires_at`; board shows active only; automatic “revoke when expired” in executor/planner is not implemented.
- **Wiring into agent loop**: Supervised loop does not yet call `evaluate()` for “may_batch” or “is_always_manual”; human_policy is additive and can be wired when desired.
- **Task/lane scope**: Policy model has project/pack; task and lane scopes are not yet used in evaluation.
- **Routing priority overrides**: `RoutingPriorityOverride` exists in config; evaluation does not yet return “prefer_lane” / “prefer_pack” to router.
- **Exception policy**: Model and config exist; evaluation does not yet use “allow_exceptions” or “granted_by_scope” for one-off exceptions.
- **Audit log**: Override apply/revoke are stored with timestamps; no separate audit log or export for compliance.
